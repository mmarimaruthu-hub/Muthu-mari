# Assumption Log

## 1. Overview
This document records all assumptions made during the design, development, and documentation of the Enterprise Patient 360 Data Platform. These assumptions underpin key architectural and business decisions and should be validated with stakeholders before production deployment.

---

## 2. Assumption Log Table

| Assumption ID | Category | Assumption Description | Impact if Incorrect | Validation Required | Status |
| :--- | :--- | :--- | :--- | :--- | :--- |
| ASM-001 | Data Model | In the absence of a unique Patient ID in the source data, the combination of `Age_Group`, `Gender`, `Race`, `Ethnicity`, and `Zip_Code_3_digits` is assumed to be a sufficient surrogate key to identify a patient cohort. | Patient records from different individuals with the same demographics will be merged together, causing incorrect lifetime aggregations. | Confirm with business/clinical team whether a Patient ID field is available or can be derived. | Open |
| ASM-002 | Data Volume | The platform is designed to handle 500M+ records with incremental processing. It is assumed that the daily incremental volume will not exceed 20% of the total record count in a single batch. | Auto Loader and MERGE performance may degrade significantly for extremely large batches. | Review with infrastructure team on cluster sizing and benchmark testing. | Accepted |
| ASM-003 | Data Format | All source data files are assumed to be in **CSV format** with a consistent header row. Multi-line fields are assumed to be quote-enclosed. | Non-CSV files or files without headers will fail the Auto Loader ingestion. | Confirm with source systems (EHR, CRM, Billing) on file export formats. | Accepted |
| ASM-004 | Schema Stability | The core schema (column names and types) in the source CSV files is assumed to be relatively stable. Minor schema drift (new columns) is handled via `cloudFiles.schemaEvolutionMode = "rescue"`. | Major schema changes (column renames, type changes) will break the Silver transformation logic. | Implement a schema change notification process with source system owners. | Open |
| ASM-005 | Financial Data | `Total_Charges` always represents the gross billed amount and `Total_Costs` represents the hospital's operational cost. The difference (`Gross Margin`) is a valid business metric. | `lifetime_gross_margin` calculations will be misleading if the definitions differ by facility or payer. | Validate with Finance/Accounting team. | Accepted |
| ASM-006 | Length of Stay | `"120 +"` in the `Length_of_Stay` field represents any stay at or beyond 120 days. It is assumed that mapping this to exactly `120` is an acceptable approximation for analytics. | Long-stay patient analytics will be slightly understated (max shown as 120 days). | Confirm with clinical analytics team that 120-day cap is acceptable. | Accepted |
| ASM-007 | Incremental Logic | It is assumed that `bronze_load_timestamp` (added by the Bronze ingestion notebook) is the correct and reliable watermark for Silver incremental processing. | If Bronze records are backdated or the timestamp is inaccurate, the Silver layer may miss records. | Ensure Databricks cluster timezone is standardized to UTC across all environments. | Accepted |
| ASM-008 | Data Freshness | The ETL pipeline is assumed to run **daily** in a scheduled Databricks Workflow. | If the pipeline runs more or less frequently, watermark-based logic may produce unexpected gaps or overlaps. | Confirm scheduling cadence with the Data Engineering operations team. | Accepted |
| ASM-009 | Gold Partitioning | Partitioning the Gold table by `encounter_discharge_year` and `patient_age_group` is assumed to match the primary query patterns of the downstream BI dashboards (filtering by year and age). | BI tools will perform full-table scans if actual query patterns differ significantly, degrading performance. | Review query patterns with the BI/Analytics team after initial dashboard development. | Open |
| ASM-010 | Source System Identifier | All data ingested is assumed to originate from **SPARCS (Statewide Planning and Research Cooperative System)**. The `source_system = "SPARCS"` literal is hardcoded in the Silver layer. | If data from other sources is later added, the `source_system` literal will be incorrect, making lineage tracking unreliable. | Parameterize `source_system` if multi-source ingestion is planned in future iterations. | Open |
| ASM-011 | PII / Privacy | The dataset is assumed to have been pre-anonymized/de-identified (Zip Code truncated to 3 digits, no direct patient name or SSN). This platform does not implement additional PII masking. | If the data is not pre-anonymized, the platform will be non-compliant with HIPAA. | Confirm data anonymization status with the Data Governance and Legal team before production deployment. | Open |
| ASM-012 | Workspace Configuration | The Databricks catalog (`workspace`), schema (`default`), and volume paths (`/Volumes/workspace/default/...`) are assumed to be pre-provisioned and accessible to the ETL service principal. | Pipeline will fail on startup with permission or path-not-found errors. | Confirmed via manual infrastructure setup prior to first run. | Accepted |
